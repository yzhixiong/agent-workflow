官网模块测试计划与测试报告规范

本文档用于把“测试”从单一清单升级为“按模块验收 + 按模块出报告”的执行机制。 当前项目由 Ocean 单人负责，因此测试计划需要与排期基线直接联动，做到模块完成一个、测试一个、报告一个、缺陷闭环一个。

测试责任人 Ocean
测试方式 模块级测试 + 上线前总回归
报告要求 每个模块输出单独测试报告
联动文档 与 23 号进度管控页同步更新

测试原则

执行原则

  • 模块开发完成后，不等待所有模块一起测试，而是立即执行该模块测试。
  • 每个模块必须形成单独测试报告，记录通过项、缺陷项和结论。
  • 缺陷关闭后需要回归，并在报告中补记回归结论。

验收原则

  • P0 模块必须测试通过并出报告，才允许进入上线候选版本。
  • P1 模块未通过不影响首期上线时，需在报告中明确标记和处理策略。
  • 上线前仍需输出一份总体验收报告，覆盖跨模块主流程。

测试报告模板字段

字段 填写要求
报告名称统一使用“模块测试报告_模块名_YYYY-MM-DD”格式。
模块名称与本页模块矩阵保持一致。
测试日期填写实际测试日期，必要时可包含回归日期。
测试责任人当前统一填写 Ocean。
版本信息记录测试对应的页面版本、接口版本或构建版本。
测试环境记录本地、测试环境或预发布环境。
测试范围说明本次覆盖了哪些页面、功能点或接口。
测试结果统计记录总用例数、通过数、失败数、阻塞数。
缺陷清单记录缺陷描述、严重级别、是否已修复、是否已回归。
结论明确写“通过 / 有条件通过 / 不通过”。
后续动作如需修复、回归或延后上线，必须写明下一步动作。

建议报告结构

建议每份模块测试报告至少包含以下结构：模块概述、测试范围、测试环境、测试结果统计、缺陷清单、回归结果、最终结论、后续动作。 如果某个模块只做了部分能力，也要在报告中明确“本次未覆盖范围”，避免默认等同于全部验收通过。

与项目进度的联动要求

联动更新

  • 模块测试报告完成后，同步更新 23 号文档中的里程碑状态。
  • P0 模块未通过时，必须在风险清单中记录并跟踪。
  • 总体验收报告通过后，才允许进入正式上线动作。

缺陷处理

  • 阻塞上线的缺陷必须先修复并回归，再更新报告结论。
  • 不阻塞上线的已知问题，需要在报告中记录影响范围和后续计划。
  • 如果某模块延期，不只改排期，还要同步调整测试窗口和报告日期。
这份文档定义的是测试机制，不是最终测试报告本身。后续进入 M3 和 M4 后，可以按本页模板继续生成具体模块测试报告。
