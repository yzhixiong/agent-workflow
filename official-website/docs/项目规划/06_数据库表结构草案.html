官网数据库表结构草案

本草案面向官网前后台一体化场景，围绕“内容管理、站点配置、线索收集、招聘管理、权限控制、文件资产”六类数据展开。 目标是先把表结构边界和关系框架定义清楚，后续再根据开发实现补充索引、约束和枚举细节。

关键关系建议

关系 说明 设计建议
角色 N:N 权限点 角色与按钮/操作权限点应支持多对多关联。 增加 sys_role_permission 关联表。
产品分类 1:N 产品 一个产品分类下可有多个产品或工艺能力条目。 category_id 外键，支持排序。
产品 1:1 产品详情 列表基础信息与详情长字段分离，便于维护。 详情表以 product_id 唯一关联。
解决方案 N:N 产品 一个方案可关联多个产品，一个产品也可出现在多个方案中。 增加 cms_solution_product 关联表。
解决方案 N:N 案例 方案页可展示多个典型案例。 增加 cms_solution_case 关联表。
案例 1:N 图集 案例详情通常包含多张展示图。 独立图集表，支持排序。
询盘 1:N 附件 客户可能上传多个需求附件。 单独附件关联表。
岗位 1:N 简历 一个岗位下存在多个投递记录。 resume.job_id 关联岗位。
文件资源 1:N 业务引用 图片、视频封面、PDF、简历、Banner 均统一走文件资产表。 各业务表只存 file_id。

通用字段建议

字段类型 建议字段 说明
主键字段 id 建议使用 bigint 或雪花 ID。
状态字段 status、is_deleted 支持启用/禁用、逻辑删除。
排序字段 sort_no Banner、分类、案例、新闻等建议统一。
多语言字段 name_zh、name_en / title_zh、title_en 优先用显式字段，简单清晰。
审计字段 created_at、created_by、updated_at、updated_by 建议所有后台维护类表统一具备。
SEO 字段 slug / seo 关联表 文章、产品、方案建议支持 URL 别名。
数据库设计建议优先保证可维护性和清晰边界，不要过早抽象成过度通用模型。 对官网这类企业站，先把核心业务对象拆清楚，比追求万能 CMS 模型更重要。 权限模型建议首期落地到“菜单级 + 页面操作级”，数据权限和字段权限暂作为后续扩展保留。
