官网数据库 ER 关系图

本文档从实体关系视角梳理官网系统的核心数据域，重点突出“内容域、线索域、权限域、资源域”的主从关系和多对多关系。 它与数据库表结构草案互为补充，前者讲关系，后者讲字段。

数据域划分

内容域

  • cms_site_config
  • cms_banner
  • cms_content_block
  • cms_product_category / cms_product / cms_product_detail
  • cms_solution / cms_case / cms_news / cms_video / cms_job

线索域

  • lead_inquiry / lead_inquiry_file
  • hr_resume
  • 与岗位、文件资源形成引用关系

权限域

  • sys_user
  • sys_role
  • sys_menu
  • sys_role_menu

资源域

  • cms_file_asset
  • 为 Banner、产品、案例、新闻、视频、简历等提供统一文件引用

核心关系表述

主实体 关系 从实体 关系说明
sys_role1:Nsys_role_menu一个角色可以配置多个菜单授权。
sys_menu1:Nsys_role_menu一个菜单可以被多个角色引用。
cms_product_category1:Ncms_product一个分类下可有多个产品或工艺能力。
cms_product1:1cms_product_detail产品主信息与详情扩展分离。
cms_solutionN:Ncms_product方案与产品为多对多，建议中间表。
cms_solutionN:Ncms_case方案详情可关联多个典型案例。
cms_case1:Ncms_case_gallery案例详情通常需要多张展示图片。
cms_job1:Nhr_resume一个岗位对应多个简历投递记录。
lead_inquiry1:Nlead_inquiry_file一个询盘可能上传多个附件。
cms_file_asset1:N多业务表文件资源作为统一引用中心，被多个业务表通过 file_id 关联。

建议补充的中间关系表

关系表 核心字段 说明
cms_solution_productid、solution_id、product_id、sort_no解决方案与产品关联表，用于方案页推荐产品。
cms_solution_caseid、solution_id、case_id、sort_no解决方案与案例关联表，用于方案页展示典型案例。
sys_user_roleid、user_id、role_id如果需要支持一个用户多个角色，建议增加该表。
cms_page_block_relid、page_code、block_id、sort_no如果单页内容块复用度高，可增加页面与内容块关联表。
官网项目的 ER 设计应尽量“清晰、稳定、可维护”。首期不建议引入过度通用的动态模型， 先以明确业务实体建模，后续再基于实际扩展需求逐步演进。
