# Standard Website Flow

适用于官网、品牌站、专题页、产品介绍站等内容表达和转化导向较强的项目。

```text
默认流程：
Orchestrator -> Analyst -> UI Designer -> Architect -> Developer -> Reviewer -> Orchestrator

阶段职责：
1. Orchestrator
   - 判断当前阶段
   - 组织上下游交付
   - 控制回退和阶段切换
2. Analyst
   - 明确项目目标、页面范围、目标用户、信息优先级和转化动作
3. UI Designer
   - 明确视觉方向、页面层级、组件风格和交互表达
4. Architect
   - 明确页面结构、组件边界、路由、数据流、工程实现方案
5. Developer
   - 按任务范围实现代码并提供改动说明和自测结果
6. Reviewer
   - 判断是否满足需求、设计和交付质量要求
7. Orchestrator
   - 汇总结论，决定进入下一轮、回退还是准备交付

阶段进入条件：
1. 需求目标、页面范围、目标用户、转化动作未澄清，不进入设计或架构阶段。
2. 设计方向、信息层级或页面重点未明确，不进入详细架构或开发阶段。
3. 页面结构、组件边界、数据来源、SEO 与性能策略未明确，不进入开发阶段。
4. 开发未提供改动说明、自测结果和已知限制，不进入审查阶段。
5. 审查未给出明确结论、问题等级和处理建议，不进入交付判断阶段。

回退规则：
1. 发现需求不清，回退给 Analyst。
2. 发现视觉方向冲突或设计不可落地，回退给 UI Designer。
3. 发现结构、组件或数据方案不清，回退给 Architect。
4. 发现实现偏差、缺陷或未覆盖场景，回退给 Developer。
5. Orchestrator 只负责编排和判断，不替代专业角色补位。

交付物要求：
1. 每个阶段都必须给出结构化结果。
2. 每个阶段都必须给出下游可直接使用的说明。
3. 每个阶段都必须显式标注风险、缺失项和待确认点。
4. 编号建议使用稳定编号，例如 FR、NFR、Page、Comp、Task、Risk、Bug。
```
